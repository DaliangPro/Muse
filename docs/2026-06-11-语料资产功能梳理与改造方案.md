# 语料资产功能梳理与改造方案

> 基线：2026-06-11 全链路精读（UI 层 19 文件 + 服务层 5 文件 + 存储层 3 文件，约 6700 行）。
> 本文是语料资产功能后续改造的指导底稿；动工时逐条认领，完成后回写状态（参照 REPAIR_PLAN 使用规则）。

## 功能现状一句话

把日常语音输入的识别记录，经「本地粗筛 → LLM 按用户规则提炼 → 规范化校验」产出五类创作素材候选（好问题/好观点/表达框架/案例素材/金句），人工逐条审核后沉淀入资产库；全程任务留痕、来源可溯。骨架完整，主要问题是**多条建好的能力没接到 UI**、**一个防重缺陷**和**一个并发写隐患**。

## 一、问题清单

状态图例：⬜ 未开始 ｜ 🟡 进行中 ｜ ✅ 已完成 ｜ ❌ 决定不修

### P0 · 数据正确性

#### 1 ✅ 跨任务无防重，候选重复堆积（核心缺陷）
- **现象**：每次提炼固定全量重提「最近 7 天」，已忽略/已入库的内容会再次变成新候选。
- **根因**：去重只在单次运行内（`AssetExtractionNormalizer` 的 `seenKeys`）；`saveCandidates` 每次都用新 UUID 插入，不对照库内既有候选/资产。
- **影响**：高频使用时待审池重复项越积越多，「忽略」失去意义（下次又来）。

#### 2 ✅ LanguageAssetStore 并发写隐患：无 WAL、无 busy_timeout、写入不检查返回值
- **证据**：`LanguageAssetStore.init` 仅 `sqlite3_open` + 建表；对照 `HistoryStore.swift:45` 已开 WAL（B5 修复时漏了这个 Store）。
- **场景**：识别主链路（HistoryStore 连接）与资产页（本 Store 连接，AssetLibraryTab 一处就建 3 个 store 实例）并发读写**同一个 history.db**；未设 busy_timeout 遇锁立即返回 SQLITE_BUSY，所有 INSERT/UPDATE 不检查结果 → **候选、资产、动作日志可能静默丢失**。open 失败同样静默（db=nil 后全部 no-op）。
- **附带**：四张表零索引（candidate.status、asset.status、created_at 均无）。

### P1 · 核心体验

#### 3 ✅ 四种提炼范围只暴露了一种
- **现象**：管线完整支持「最近 7 天 / 最近 30 天 / 最近 N 条 / 手动勾选记录」（`AssetExtractionConfiguration` 四个工厂方法），UI 写死 `last7Days(maxRecordCount: 50)`（`AssetLibraryTab.swift:351`）。其余三种是建好没接线的死代码。

#### 4 ✅ 资产库只进不出：无删除、无收藏，误入库不可撤销
- **现象**：资产详情只有「复制」。数据层 `isFavorite` 字段、`favorited/unfavorited/deleted` 动作类型、软删状态 `archived/deleted` 全部就绪，UI 零入口。

#### 5 ✅ 候选忽略不可逆
- **现象**：误点「忽略」无法找回；数据还在（status=ignored），只是没有任何视图能看到。

#### 6 ✅ 提炼中无法取消、无超时
- **现象**：LLM 调用无超时设置；提炼期间按钮只显示进度，长时间挂起只能等。管线已是 async/actor 架构，取消接线成本低。

### P2 · 体验细节

#### 7 ✅ 提炼范围与规则对用户不可见
- 点「提炼」前不知道会提哪些记录、按什么规则；提炼后也只有 Job 摘要一行。

#### 8 ✅ 「今日发现」名不副实
- 展示的是历史累积的全部 pending 候选，并非今日产出；与最近一次提炼任务也无视觉关联。

#### 9 ✅ 1.2 万字符静默截断
- `maxTotalInputCharacters: 12_000` 超出的记录直接丢弃，无任何提示；用户不知道哪些输入没被看过。

#### 10 ✅ 本地寒暄过滤形同虚设
- `isLikelyLowValue` 用**全等匹配**短语表（"好的好的"、"嗯嗯"都漏过）；实际靠 LLM 兜底，白白消耗 token。

#### 11 ✅ 概览页计数全表加载
- `GeneralSettingsTab` 用 `fetchAll().count` 数资产条数，应为 `COUNT(*)`。

### P3 · 技术债清理

#### 12 ✅ 遗留类型与旧格式兼容层
- `term/snippet` 类型、`ProviderResponse` 的 viewpoints/quotes/terms/snippets 旧字段解析。需确认无老库数据依赖后删除。

#### 13 ✅ 无人调用的旧入库路径
- `AssetExtractionNormalizer.normalize()`（直接产 LanguageAsset）无调用方；`AssetExtractionRunResult.assets` 恒为空数组。

#### 14 ✅ 候选表无限增长
- saved/ignored 候选永久留存无清理；叠加问题 1 的重复堆积，表膨胀更快。

#### 15 ✅ 「资产→工作流」三动作有统计无功能
- `addedToHotwords / addedSnippetDraft / generatedPromptDraft` 动作类型与统计口径俱在，全无 UI——资产沉淀后没有下游出口（只能复制）。归在 P3 是因为先要产品决策：做（接常用词）还是砍（删死代码）。

## 二、逐条解决方案

| # | 方案 | 要点 |
|---|------|------|
| 1 | 防重三件套（一期做前两件） | ① 取数阶段排除已被既有候选/资产引用过的 `source_record_ids`（查两表的 ids 集合做差）；② Normalizer 去重键对照库内已有候选+资产（type\|title\|content 小写键，提炼前一次性加载成 Set）；③（可选二期）Prompt 附带已有资产标题清单让模型主动避开——token 成本高，先不做 |
| 2 | 照搬 B5 的 HistoryStore 修法 | init 补 `PRAGMA journal_mode=WAL` + `sqlite3_busy_timeout(db, 3000)`；为 candidate(status)、asset(status)、两表 created_at 建索引；open 失败与 step 失败打日志；HistoryStore 同样补 busy_timeout（当年只开了 WAL） |
| 3 | 工具栏「提炼」旁加范围选择 | 下拉四项：最近 7 天 / 最近 30 天 / 最近 N 条 / 手动选择（弹识别记录勾选列表）；记忆上次选择（UserDefaults）；Job 表已有 range 字段，无 schema 改动 |
| 4 | 资产详情补「收藏 / 删除」 | 删除走软删（status=deleted，留 action_log），列表加收藏筛选/置顶；数据层零改动，纯 UI 接线 |
| 5 | 加「已忽略」筛选入口 | 今日发现视图加状态切换（待审 / 已忽略），已忽略项提供「恢复待审」；`updateCandidateStatus` 已支持 |
| 6 | 超时 + 可取消 | LLM 调用包 `withTimeout`（建议 120s）；提炼中按钮变「取消」，取消时把 Job 标记 failed("用户取消")；`runExtraction` 的 Task 持引用即可 cancel |
| 7 | 提炼前预览 | 点提炼先本地跑取数+过滤（零成本），弹确认：「将提炼 N 条记录（范围 X，规则：仅A·强过滤），预计输入 M 字」；同时解决问题 9 的截断提示（超量时此处明示） |
| 8 | 改名 + 按任务分组 | 「今日发现」→「待审候选」；列表头部显示最近一次提炼任务摘要（时间/范围/产出数），新产出候选标「新」徽章 |
| 9 | 并入方案 7 | 截断信息进预览弹窗 + Job summary 写明「超量丢弃 M 条」 |
| 10 | 判定逻辑改包含式 | 去标点后长度 <6 且仅由语气词字符集构成才过滤；或删除该层（拍板项，推荐改判定保留——省 token） |
| 11 | 加 `COUNT(*)` 查询 | LanguageAssetStore 加 `func count(status:)`，概览页改用 |
| 12 | 确认后删除 | 先查线上库有无 term/snippet 数据（一条 SQL），无则连类型带兼容层删除；有则保留读取兼容、删除写入路径 |
| 13 | 直接删除 | `normalize()` 与 `AssetExtractionRunResult.assets` 字段，git 可追溯 |
| 14 | 保留策略 | saved/ignored 候选保留 90 天或 1000 条（取先到），提炼成功后顺手裁剪（沿用历史库裁剪模式） |
| 15 | 产品拍板项 | 方案 A：一期接「加入常用词」（资产→热词，与识别质量正反馈闭环，推荐）；方案 B：砍掉三动作与统计死代码。待用户决策 |

**建议施工顺序**：P0（1、2，数据正确性优先）→ P1（3、4、5、6）→ P2（7+9 合并、8、10、11）→ P3（12、13、14、15 拍板后顺手）。P0 两条互不依赖可并行；方案 7 依赖方案 3 的范围选择先落地。

## 三、改造后的功能蓝图

**提炼：从黑盒按钮变成可控流程。** 用户点「提炼」，先选范围（7 天/30 天/最近 N 条/手动勾选），看到预览——「N 条记录将被提炼，规则：仅 A、强过滤，重点人群：内容创作者」，确认后执行；过程可随时取消；每次产出**只有新内容**，忽略过的不再阴魂不散。

**审核：可反悔的待审池。** 「待审候选」按最近任务分组，新产出带标识；看走眼忽略了，切到「已忽略」一键恢复。审核动作轻、心理负担小——因为一切可逆。

**资产库：有完整生命周期。** 入库的资产可收藏置顶、可软删除、可复制，（若拍板方案 A）高频术语类资产可一键加入常用词，反哺识别准确率——语料资产从"只读陈列柜"变成"创作工作流的中转站"。

**底座：数据不丢、库不膨胀。** 资产库连接与识别主链路并发安全（WAL+忙等待+索引）；候选表有保留策略；死代码清空。概览页那盏「语料资产 N 条」的灯，数字背后是一条条真正不重复、可追溯、有去处的创作素材。
