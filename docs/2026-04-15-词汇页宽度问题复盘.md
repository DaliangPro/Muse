# 2026-04-15 词汇页宽度问题复盘

## 结论

这次问题的根因，不是外层白卡片的统一宽度失效，而是 `片段替换` 卡片内部某一行存在过强的固有宽度，导致整张卡片被撑宽。

真正的撑宽源是：

- `snippetAddRow`
- 其中的 `替换内容`、`触发词`、箭头、`添加片段` 按钮原本是固定宽度组合
- 这组内容的总固有宽度超过了目标卡片宽度，最终把下方卡片整体向右撑出

最终修正方式：

- 保持外层卡片宽度逻辑不变
- 只把 `snippetAddRow` 里的 `替换内容` 输入框改成可伸展布局
- 让这一行不再以固定宽度总和去决定卡片宽度

## 这轮学到的经验

### 1. 先看真实窗口，再看代码

不能先凭代码猜。

正确顺序应该是：

1. 打开真实设置窗口
2. 截取单独窗口，不用全屏脑补
3. 肉眼确认问题确实存在
4. 再回到代码定位参数和布局关系

### 2. 要把“容器层”和“内容层”分开检查

这类问题至少有三层：

- 外层白卡片宽度
- 内层灰色内容区宽度
- 某一行内容自己的固有宽度

前两层一样，不代表最后视觉一定一样。
如果某一行用了多个固定宽度控件组合，它完全可能把整张卡片撑大。

### 3. 不要再用口头判断“看起来像”

这次证明了，单靠口头描述很容易误判。

后续应该统一使用同一套验证方法：

- 同一个窗口截图
- 同一组横向采样行
- 同一套像素阈值
- 对比上下卡片左右边界

这样能直接知道：

- 是不是卡片本身变宽了
- 还是只是内容视觉轴线不同

### 4. 固定宽度组合是高风险点

以后遇到这几种写法，要优先怀疑它们是否在撑布局：

- 多个 `.frame(width: ...)` 连在一行
- 固定宽按钮 + 固定宽输入框 + 图标组合
- `fixedSize(horizontal: true, vertical: false)`
- 文案更长但仍放在同一行

这次就是典型案例：

- 外层卡片理论宽度一致
- 但 `snippetAddRow` 的固有宽度组合超过预算
- 所以出现“下卡片更宽”

### 5. 修复要先打最小补丁

正确做法不是一上来重构整页，而是：

1. 找到真实撑宽源
2. 只改那一处
3. 重新构建
4. 截同一类窗口图再验证

这次最小补丁是：

- 仅将 `snippetAddRow` 中的 `替换内容` 输入框改为 `vocabularyFlexibleTextField`

## 后续同类问题排查清单

以后再遇到“明明参数一样，但视觉不一样”，按这个顺序查：

1. 先抓真实窗口图
2. 确认问题是外卡、内卡，还是单行内容
3. 查是否有固定宽度组合
4. 查是否有 `Spacer` 把内容推偏
5. 查 footer / header 是否有超长单行内容
6. 修复后用同一套截图和像素方法复测

## 这次最终有效的方法

- 用 `screencapture -l <window_id>` 抓单独窗口
- 用同一套行坐标做边界检测
- 先证明“下卡确实更宽”
- 再证明“修复后上下边界一致”

这套方法后续可以直接复用，不需要再靠争论和猜测推进。
