# Muse 设置窗口信息架构重构方案

## 目标

把当前“模块配置表单”式设置页，重构成“状态总览 + 任务导向配置”的设置窗口。

用户打开设置页后，应在 5 秒内回答清楚 4 个问题：

1. 现在能不能正常用
2. 当前默认是怎么工作的
3. 模型和权限是否配置完整
4. 要改某一类行为，应该去哪里

## 当前主要问题

### 1. 栏目按内部实现分组，不按用户任务分组

当前左侧栏目：

- 通用
- 模型
- 词汇
- 模式
- 历史
- 关于

问题：

- 对开发者清楚，对普通用户不够清楚
- “模型”和“模式”边界模糊
- “通用”里混着系统权限、录音行为、系统集成、高级设置，范围过大

### 2. 缺少状态总览页

用户第一眼看不到：

- 麦克风 / 辅助功能是否已授权
- 当前默认模式是什么
- 当前 ASR / LLM 是什么
- 是否有更新
- 是否还有未完成配置

### 3. 高级配置和平常配置混在一起

例如：

- 启动项
- Dock 图标
- 剪贴板注入
- 代理绕过
- 热词 / 替换片段
- 模型 API 配置

这些内容的重要性完全不同，但现在都被并排当成同一层表单。

### 4. 页面组织更像“后台配置”，不像“高频桌面工具设置中心”

这会导致：

- 用户知道能改什么，但不知道该先看什么
- 页面可用，但不形成清晰的操作路径

## 重构原则

### 1. 先总览，后细节

先让用户看到当前系统状态，再进入具体配置。

### 2. 按任务组织，而不是按代码模块组织

用“输入与录音”“模型与服务”“文本处理”这种语言，而不是把内部实现名直接暴露给用户。

### 3. 基础设置前置，高级设置后置

高频项优先展示，低频项折叠或放到更深层级。

### 4. 一个页面只解决一类问题

避免一个页面同时回答 5 种不同类型的问题。

## 新的信息架构

### 左侧一级导航

推荐调整为：

1. 总览
2. 输入与录音
3. 模型与服务
4. 文本处理
5. 词库
6. 历史
7. 关于

## 各栏目定义

### 1. 总览

作用：打开设置页后的首页，只回答“现在是什么状态”。

包含内容：

- 系统状态卡
  - 麦克风
  - 辅助功能
  - 当前更新状态
- 当前工作方式卡
  - 默认输入模式
  - 当前 ASR 引擎
  - 当前 LLM 引擎
- 快速入口卡
  - 去配置权限
  - 去配置模型
  - 去配置模式

不放：

- 大量细节表单
- 冗长帮助文案

### 2. 输入与录音

作用：处理“怎么录、怎么触发、怎么表现”的问题。

包含内容：

- 快捷键
- 提示音
- 录音动效
- ESC 中断
- 录音时降低音量

从当前页迁移来源：

- 通用设置中的“录音行为”
- 模式页中和快捷键直接相关的内容

### 3. 模型与服务

作用：处理“识别和处理引擎怎么配置”的问题。

包含内容：

- ASR 引擎
- LLM 引擎
- 本地 / 云端状态
- 测试连接
- 凭据保存

从当前页迁移来源：

- 模型页

但组织方式调整为：

- 先显示“当前已启用什么”
- 再显示详细配置表单

### 4. 文本处理

作用：处理“识别之后文本怎么被整理和输出”的问题。

包含内容：

- 默认模式
- 各模式说明
- 文本后处理策略
- 注入剪贴板
- 代理绕过

从当前页迁移来源：

- 模式页
- 通用页的部分高级设置

### 5. 词库

作用：处理热词和文本替换。

包含内容：

- ASR 热词
- 替换映射
- 片段替换

从当前页迁移来源：

- 词汇页

### 6. 历史

作用：历史记录、日志、保留策略。

包含内容：

- 会话历史
- 搜索/筛选
- 保留策略

从当前页迁移来源：

- 历史页

### 7. 关于

作用：版本、许可证、支持、更新。

包含内容：

- 当前版本
- 检查更新
- 更新状态
- 开源许可证
- 支持信息

从当前页迁移来源：

- 关于页

## 首页总览的具体结构

推荐首页采用三段式：

### 区块 1：当前状态

- 麦克风：已授权 / 未授权
- 辅助功能：已授权 / 未授权
- 更新：已是最新 / 有可用更新

### 区块 2：当前默认工作方式

- 默认模式
- ASR 提供商
- LLM 提供商

### 区块 3：下一步操作

- 去完成权限
- 去配置模型
- 去调整录音体验

## 不建议继续保留的组织方式

### 不建议继续把“通用”做成大杂烩

因为它会无限膨胀。

### 不建议让“模型”和“模式”并列但解释不清

用户很难一眼理解：

- 模型是底层引擎
- 模式是工作流程

这两个应该通过页面标题和说明明确拆开。

### 不建议让“关于”承担太多功能

更新检查属于产品状态，不只是“关于”。
如果保留在“关于”，首页也应同步显示更新摘要。

## 推荐的重构顺序

### 第一步：重做信息架构

不动视觉，只调整：

- 左侧导航名称
- 页面归属
- 首页总览逻辑

### 第二步：重做页面骨架

做出：

- 新的左侧导航
- 新的总览页
- 新的各页标题结构

### 第三步：统一视觉语言

最后再统一：

- 卡片层级
- 控件样式
- 色彩节奏
- 页面头部

## 本轮建议结论

当前最值得做的不是直接美化，而是：

1. 把设置窗口从“表单集合”改成“状态总览 + 任务导向配置”
2. 新增“总览”页作为入口
3. 把“通用 / 模型 / 模式”重组为更符合用户心智的栏目
