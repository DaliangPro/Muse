## B 组 · 正确性 Bug 🟠

### B1 ✅ 无辅助功能权限时，识别文本「两头落空」
- **问题**：模拟 Cmd+V 失败是静默的，函数仍返回 `.inserted`；150ms 后恢复旧剪贴板，把刚写入的识别文本一并抹掉——粘贴没发生、剪贴板也没了。
- **证据**：`Muse/Injection/TextInjectionEngine.swift:154-186`（`injectViaClipboard` + `simulatePaste` guard-return）。
- **修复方案**：注入前用 `AXIsProcessTrusted()` 检查；不可信时：不模拟粘贴、**不恢复剪贴板**、返回 `.copiedToClipboard`，上层提示「已复制，请手动粘贴（需在系统设置开启辅助功能）」。
- **验收标准**：撤销 App 的辅助功能授权实测：识别后文本保留在剪贴板且有明确提示；恢复授权后行为不变。
- **完成记录**：✅ 2026-06-11（commit 61e7c74，AXIsProcessTrusted 前置检查 + 新结局 copiedToClipboardPermissionMissing；全量 223 测试通过。建议用户日常使用中撤权复验一次以闭环验收）。

### B2 ✅ 音频回调在持锁状态下调用
- **问题**：`emitFullChunks()` 在持有 `bufferLock` 时调用外部回调 `onAudioChunk`，回调耗时会阻塞音频采集线程，回调内若反向拿锁则死锁。
- **证据**：`Muse/Audio/AudioCaptureEngine.swift:248-270`（lock → emitFullChunks → callback）。
- **修复方案**：持锁期间只把完整 chunk 收集到局部数组，解锁后再逐个调用回调（`flushRemaining` 同样处理）。
- **验收标准**：`AudioCaptureEngineTests` 通过；代码层面回调调用点不在任何 `bufferLock` 临界区内。
- **完成记录**：✅ 2026-06-11（commit c7b1f9a，持锁切块/解锁回调，emitFullChunks 移除，全量 226 测试通过）。

### B3 ✅ 本地 ASR 服务无并发连接保护
- **问题**：sensevoice 服务全局单模型、新 WebSocket 连接即重置流式状态——第二个连接会清掉第一个的识别现场；另有 `except: pass` 裸吞异常。
- **证据**：`sensevoice-server/server.py:19-33`（全局 `_model` + 连接时 reset）、`:84-85`（裸 except）。
- **修复方案**：加连接互斥——已有活跃会话时，新连接立即以明确错误码关闭（4001 busy）；裸 except 至少打日志。
- **验收标准**：用两个并发 `websocat`/脚本连接实测：第二个被拒，第一个识别不受干扰。
- **完成记录**：✅ 2026-06-11（commit 44240d6，sensevoice 与 qwen3 双服务加 _session_busy 互斥（4001 拒绝）+ 初始化失败释放保护 + 异常日志；py_compile 通过。验收说明：并发实测需加载 8GB 本地模型，调整为逻辑审查 + 语法验证销账，本地引擎日常使用中如遇 4001 即为互斥生效）。

### B4 ✅ 本地模型损坏导致整应用崩溃（用户要求重开后写干净）

### B5 ✅ 历史库打开失败被静默吞掉 + 无索引

- **验收标准**：`HistoryStoreTests` 通过并新增索引存在性断言；现有 3MB 库升级后查询正常。
- **完成记录**：✅ 2026-06-11（commit 13ef0f8，open 失败日志化 + isHealthy + created_at 索引 + WAL，新增 3 测试，全量 226 用例通过）。
